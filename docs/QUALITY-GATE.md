# MultiClaw Quality Gate

This quality gate keeps AIML Solutions repos professional, marketable, and honest. It is designed for public portfolio work, client handoff, and recruiter review.

## Portfolio-Ready Criteria

| Criterion | Standard |
| --- | --- |
| Purpose | README explains who the repo is for and what problem it solves |
| Evidence | At least one test, command, artifact, or metrics document supports the claims |
| Status | Implemented, scaffold, and planned work are clearly separated |
| Safety | Secrets, private account data, paid-task details, and live financial actions are excluded |
| Commercial fit | README includes hiring, client, or operational use cases |
| Maintenance | Next improvements are concrete and small enough to ship |

## Language Standard

Use confident, specific language:

- "validation-first data pipeline" instead of "demo script"
- "local-first deployment pattern" instead of "toy Kubernetes example"
- "AI harness artifact" instead of "prompt notes"
- "source readiness smoke check" instead of "quick API test"

Avoid inflated language:

- do not claim production scale without production evidence
- do not claim trading performance without controlled backtest methodology
- do not claim enterprise deployment without deployed environment proof
- do not claim private client or platform results in public repos

## Release Checklist

Before pushing a public repo update:

1. Run the smallest relevant test or validation command.
2. Check that README claims match implemented evidence.
3. Confirm no secrets, credentials, private task names, or account IDs are included.
4. Keep examples generic but realistic.
5. Record a resume/client-ready proof bullet if the update strengthens positioning.
